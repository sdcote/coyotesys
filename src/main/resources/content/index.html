Products

Whether running on embedded SBCs in the field, hosted in the cloud or your own data center, Coyote Systems has the data integration solution components to meet your needs. Chose only the toolkits needed for your particular integration.

CoyoteDX

Core data exchange toolkit and framework for creating any number of managed integrations.

CoyoteFT

File Transfer tools supporting the transfer of files across FTP and SFTP. Great for pre and post processing tasks to stage data and deliver results.

CoyoteSN

Send and receive data from any ServiceNow instance. Uses the table model API to access any data in the SaaS instance and exchange it with any other system.

CoyoteUI

Allows for the support of an optional Web UI to directly manage the data exchange service and all the jobs it supports.

CoyoteMT

Send and receive data through SMTP, IMAP and POP3 services. Tools include components for sending emails and SMS texts on events in the data exchange jobs and their results.

CoyoteWS

Send and receive data through any JSON/XML web service. Tools include support for SOAP and even screen scraping.

CoyoteSNMP

Read and write SNMP traps and events. Makes anything SNMP accessible and allows for automated monitoring and management of SNMP components. Allows an integration to not only communicate with SNMP devices, but to be monitored and managed by enterprise NMS (network monitoring system).

CoyoteMQTT

Send and receive MQTT messages to any standardized broker. Supports message caching and SSL/TLS connections. Perfect for field deployment in embedded SBCs to communicate with head-end or SCADA systems.

CoyoteDF

Data Factory tools to generate randomized data using any number of strategies. Great for Load/Stress testing in Continuous Integration (CI) pipelines or general exploratory testing.

CoyoteMB

Highly resilient components communicate with a peer to peer (server-less) packet protocol to coordinate work, monitor peers and enable OA&M capabilities for any application on any platform, from SBC to cloud deployments.

Features / Capabilities

Coyote DX offers many capabilities previously found only in expensive, monolithic packages.

Security

Confidentiality, Integrity and Availability are built into each component and drive all design and technical decisions and approaches. Coyote DX was developed in the telecommunications and electric utility industries where security is of prime importance. NERC-CIP, NIST SP-800, NIST IR 7628, SOX, HIPPA, DHS Procurement Language all influenced DX design.

Portability

Runs on *nix and MSwin hosts, in local data centers, in all popular cloud platforms and on embedded SBC devices. Coyote DX is a 100% Java application which means it will run on any system supporting Java. From tiny SBC devices to massive mainframes, Coyote DX can be deployed anywhere.

Modifiable, Off The Shelf

You have access to all code. It's yours. Extend the product as you see fit. Build in your own proprietary capabilities and keep it separate from the rest of the product. Whether you add new features, have us do it or choose a third-party, you are in complete control. No vendor lock-in here!

Modularity

Deploy only the components you need. Develop your own. Each component can live in its own library and dropped into the deployment. Just change the job configuration file and Go!

Monitoring

Every component is completely monitor-able if you so desire. Instrumentation (i13n) is built-in and accessible via secure HTTP. SNMP is even available for monitoring by infrastructure management tools.

Management

Secure HTTP web services are built in to the core loader which can be enabled to provide ReST services to manage any component of the system. SNMP can even be used to set custom OIDs changing the way the components operate at runtime. Custom management protocols can be implemented to mee your unique needs.

Simplicity

All components are designed with simplicity as a goal. SOLID and DRY principles were used to ensure everything fits together with minimal effort. Simple to deploy, simple to operate, simple to maintain, simple to expand. Simplicity keeps costs low even after deployment.

Scalability

With the core weighing in at less than 800K, Coyote DX can be deployed on nearly any platform. It runs easily on laptops, SBC devices and large hosts. The exact same product running in the Amazon and Azure clouds will run on your 5 year old laptop without modification.

Expandability

Deploy only the modules you need. Need more capabilities? Add more components. Build your own. Find other components built by others from the growing community of developers writing components to support new protocols, data models, applications and perform transformations and analytics on data.

Pluggable Analytics

The modular design of the Coyote DX toolkit allow any number of analytic "listeners" to observe the data passing through the data exchange job. This enables analysis of all data and the execution of business logic when data breaches any user defined thresholds, including rate of change and throughput.

Batch Mode

Jobs can be run in "batch" mode at regular intervals using time-tested polling patterns. External schedulers such as "cron", "at" and any other job execution mechanism can run Coyote DX jobs or the internal scheduler can be used.

Real Time Mode

There are many components which run continually, processing information as it is becomes available. Coyote DX jobs can listen to message topics and queues, for example, performing processes in "real time", in contrast to "batch" mode.

Tasks

Jobs are not limited to Read-Transform-Write (RTW) operations. A job can be defined with one or more tasks to be performed in sequence. For example, a Job can be run to provision a service with a set of variables detailing the service. Password reset jobs can be triggered on behalf of users.

SaaS Mode

Run your jobs in our SaaS offering to avoid all hosting and operations costs. Run our SaaS offering in your own datacenter to give the entire organization access to data mediation capabilities. Just sign-in and submit your job in our Amazon Cloud deployed solution.

Device Convergence

Mediating the data models, communications protocols and operation activities of devices from many different vendors is challenging and complicated. Use device convergence to bring different devices from different vendors under one common model.

Service providers find themselves with a variety of devices deployed in the field. Not only different devices, but different vendors and different versions and models from the same vendor. It seems every device has its own protocol, data model, API and exchange pattern. The challenge is how to represent the different devices and their data in one unified model.

A successful approach is the use of device mediation components to abstract the vendor-specific details away from the system. CoyoteDX lives between the field devices and the data center, communicating with the devices in their own native protocols and transforming that data into one the rest of the organization understands. Data passes through the Perimeter Firewall with the devices native protocols but enters the Application Firewall using a single, uniform protocol thanks to the mediation of the CoyoteDX components.

A group of workers running in the perimeter network receive jobs via work queues to poll devices in the field and each head-end system. The various forms of data are placed in a common data format and send to other message topics/queues for back office systems to receive and process. As the number of devices in the field increase, so to do the number of workers (and message queues/topics) to accommodate the load. Workers mediate between field devices and all the various protocols and the common unified model in the corporate data center or cloud infrastructure.

Dedicated modules are written to interact with either the device directly or an intermediate (head-end) system and exchange data to the rest of the system using a common model. Voice and Data provisioning does not have to embed vendor-specific logic to interact with the hardware. Distribution management systems (DMS) can use IEC common information models (CIM) to interact with grid components using the device mediation modules as a facade to different hardware platforms.

The result is vendor lock-in becomes a thing of the past, as CoyoteDX allows any equivalent device from any vendor to be deployed where appropriate and even replace existing devices from other vendors. You can implement Demand Management with devices from multiple vendors and let CoyoteDX handle protocol conversions. Grid Sensors from any vendor can be deployed without having to change any of the existing business logic. CoyoteDX ensures each vendor sends the same set of events and metrics so the back office can concentrate on just one model.

ITSM Integration

ServiceNow makes it easy to get data in, but getting your data out is a bit limited.

ServiceNow offers the MID server which is great for getting data into ServiceNow, but not so handy for getting it out. Sure, you can export CSV and a few other formats, but getting that data loaded into other systems is a bit "hit-or-miss" as not all systems deal with these formats very well. The result is usually a manually intensive process which prone to error and not very reliable. Even making web service calls from ServiceNow into other systems is difficult and requires configuration of not only ServiceNow, but networking (e.g. firewall routes) and the target system to support the ServiceNow call. The MID server is realistically only half of an integration solution and the rest of the options require a bit of work.

It is also worth mentioning that many security groups have spoken up about placing an application in the data center under the direct control of an external system. Security audits have flagged this approach for closer scrutiny and suggest all communications are initiated and managed from within using tools under direct control of operations. Allowing an external agent or proxy behind the firewall such as the MID Server may violate security policies and require a more traditional integration approach.

Running CoyoteSN tools in CoyoteDX gives you the ability to implement bidirectional integrations with a few lines of a configuration file. You can make web service calls directly into the ServiceNow API and pull any data from any table and direct it to one or more target systems through not only flat files (CSV, XML and Fixed Field Length) but also through direct JDBC database and customized web service calls. You can even use other CoyoteDX modules to transform and write your ServiceNow data to other formats and systems.

It is now possible to safely exchange data with external business partners without having them directly access your ServiceNow instance. Easily pull data from ServiceNow tables and format it to your business partner specifications and send it to them using their native protocols. They don't need to interface with ServiceNow at all. You can also query your external partner data using their API and protocols to create records for posting to your ServiceNow instance meeting the needs and specification of your table structures. Bidirectional business partner integrations to ServiceNow using one tool is simpler and more reliable than passing CSV files around.

CoyoteSN allows you to implement integrations with ServiceNow in a few hours, not a few weeks (or months) and can be easily changed via configuration files which can be version controlled and managed by the operations staff. The developers only need to be involved when custom components are required.

Integrations As A Service

Run a pool of workers in your data center to scale integration horizontally.

Coyote Systems runs multiple workers in our cloud-based data center to handle read-transform-write (RTW) processing which scales a s demand changes. Each worker retrieves a job assignment from the job queue, pulls the job configuration and context data from cloud storage, instantiates a transform engine, then runs the transform configuration. Pre and Post processing tasks ensure the environment are initialized and cleaned-up when the job is run ensuring each worker starts with a pristine environment with no data leakage.

Data transfer jobs can be entered through the website or imported directly into the data store. These are the exact same jobs users created and ran locally; no modifications are required to run them in the cloud. This means each team can develop and debug their jobs locally and deploy them to the "cloud" when they are ready to productionalize their data transfers.

You can run a similar configuration in your data center with your own infrastructure, use our cloud, or your own cloud infrastructure, Coyote is cloud-agnostic and portable across a variety of hosting environments. This means you can provide secure data mediation services to the entire organization with one solution. Each data transfer job can be owned and maintained by different people or centrally administered.

ITIL/ITSM Automation

If you have ServiceNow, CoyoteDX can operate as and agent for a wide variety of automation tasks.

ServiceNow plug-ins can only take you so far. Sure, you can make web service calls out from business rules and scripts, but these facilities were never designed for heavy automation. What is often needed is an agent to be assigned work and have it perform all the complex processing. CoyoteDX can regularly query ServiceNow instances for work assigned to an individual or group, process it using purpose-written code to perform complex processing and update that ticket with the results.

With CoyoteDX as an agent, ServiceNow can orchestrate any number of complex tasks in the enterprise. Once an incident or task is assigned to a pool of CoyoteDX agents, it can perform its work without having to embed business logic in the ServiceNow instance. An as a separate agent, CoyoteDX agent can run anywhere outside the ServiceNow instance on any platform and in any numbers necessary to support the workload. Too many provisioning tasks on Monday mornings? - Ramp up the number of CoyoteDX ServiceNow agents handling the increased workload. Another system needs to trigger the same processing? - Either have it open an task in ServiceNow or Have another CoyoteDX agent read from that other system for work assignments and perform the work in concert with ServiceNow assignments.

CoyoteDX agents can also generate incidents in ServiceNow which allows any CoyoteDX process to report its status directly to ServiceNow for assignment to staff when processing notices exceptions or when resources are running low. Are there connection problems to a database required for an integration? CoyoteDX can open an incident so the issue can be resolved. A CoyoteDX agent may be tasked with scanning logs for patterns and opening an incident when patterns have been observed. Scanning LDAP logs for strange addresses or numerous failed authentications can trigger intrusion detection incidents. Scanning DHCP logs for suspicious activity can help detect foreign devices entering the network. The possibilities are limited only by your imagination.
